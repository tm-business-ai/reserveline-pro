# ReserveLine Pro 商品化計画

## 1. 現在できていること

ReserveLine Proは、LINE予約受付と店舗側の予約管理を想定した基本機能を備えています。現時点では、LINE公式アカウントと本番接続する前のデモ運用・検証に向いた状態です。

- LINE予約を想定した受付処理があります。お客様からのテキストメッセージを受け取り、メニュー確認、予約登録、予約確認、キャンセルの流れを確認できます。
- 管理画面があります。Streamlitの画面から予約状況を確認し、店舗側で予約を管理できます。
- 本日の予約確認ができます。当日の予約一覧を管理画面で確認できます。
- 今週の予約確認ができます。今日から7日間の予約を一覧で確認できます。
- 電話・店頭予約の登録・変更ができます。LINE以外で受けた予約を店舗側が空き時間から登録し、既存予約も変更できます。
- キャンセル管理ができます。予約ステータスを変更し、キャンセル済みの予約を一覧で確認できます。
- 予約メニュー設定があります。メニュー名、所要時間、料金、表示順、受付状態を管理できます。
- 予約データのダウンロードができます。予約一覧をCSVとして保存できます。
- 予約前のお知らせ予定を確認できます。予約前に通知対象となる予約を抽出し、デモ送信処理を確認できます。
- SQLite保存に対応しています。予約データとメニュー設定をローカルDBに保存します。
- テストがあります。予約登録、キャンセル、重複防止、営業時間外チェック、営業日チェック、Webhook処理などの基本テストがあります。
- GitHub公開時の安全対策があります。APIキー、LINEアクセストークン、チャネルシークレット、管理者パスワードは環境変数から読み込む構成です。

## 2. 商品化に足りないこと

現時点のReserveLine Proは、商品化前のMVPとしては機能が揃っていますが、小規模事業者へ実運用で提供するには、以下の強化が必要です。

- LINE公式アカウントとの本番接続: Webhook署名検証と返信処理の土台はありますが、実アカウントでの疎通確認、エラー時の対応、運用手順の整備が必要です。
- LINE上でのボタン式予約: 現在はテキスト入力中心です。実利用ではFlex Messageやクイックリプライで、メニュー、日時、確認、キャンセルをボタン操作にする必要があります。
- 予約変更機能: 現在は確認とキャンセルが中心です。お客様がLINE上で日時やメニューを変更できる機能が必要です。
- 予約前のお知らせ通知の本番送信: 現在は通知対象の抽出とデモ送信が中心です。LINE Push Messageで実際に通知する処理と送信失敗時の再試行が必要です。
- 営業日・営業時間・予約枠設定の管理画面対応: 現在は環境変数で営業日・営業時間を設定します。店舗担当者が管理画面で変更できる必要があります。
- 顧客別予約履歴: LINEユーザー単位・顧客名単位で過去予約、キャンセル、来店履歴を確認できる機能が必要です。
- 管理者認証の強化: 現在は単一パスワード認証です。本番ではセッション管理、パスワード強度、アクセス制限、必要に応じた複数管理者対応が必要です。
- データバックアップ: SQLiteファイルの定期バックアップ、復元手順、障害時の運用ルールが必要です。
- エラー・ログ管理: Webhook失敗、LINE API失敗、予約登録失敗、通知失敗を追跡できるログ設計が必要です。
- デプロイ手順: HTTPS公開、環境変数設定、DB永続化、起動プロセス、監視、再起動方法を含む本番手順が必要です。
- 本番運用時の注意点: 個人情報保護、アクセス権限、LINE公式アカウント設定、無料ホスティングの制限、障害時連絡などを明確にする必要があります。
- 画面文言の品質確認: 現在の表示文言には文字化けして見える箇所があるため、商品化前に日本語表示を整理する必要があります。

## 3. 商品化レベルの完成条件

v1.0として商品化できる状態は、以下を満たす状態とします。

- お客様がLINEで予約できる。
- お客様がLINEで予約確認できる。
- お客様がLINEでキャンセルできる。
- お客様がLINEで予約変更できる。
- 店舗側が管理画面で予約を確認できる。
- 店舗側が管理画面で電話・店頭予約を登録できる。
- 店舗側が予約メニューを設定できる。
- 店舗側が営業日、営業時間、予約間隔、予約可能枠を設定できる。
- 営業時間外、定休日、過去日時、重複予約を防止できる。
- メニュー所要時間を考慮して、予約枠の重なりを防止できる。
- 予約前のお知らせをLINEで送れる。
- 通知送信の成功・失敗を確認できる。
- 管理者以外が管理画面を操作できない。
- 予約データをバックアップ・復元できる。
- エラー発生時に原因を追跡できるログが残る。
- 機密情報がGitHubに含まれない。
- README通りに起動できる。
- `python -m compileall app ui` が成功する。
- `python -m pytest -q` が通る。

## 4. 実装ステップ

### Step 2：営業日・営業時間・予約枠設定の本格化

何を作るか:
営業日、営業時間、予約間隔、予約可能枠をアプリ内の設定として扱えるようにします。メニュー所要時間を考慮し、予約枠の重複を防ぐ基礎を固めます。管理画面では現在の予約ルールを確認できるようにします。

完了条件:
管理画面で予約ルールを確認でき、過去日時、営業時間外、定休日、同一時刻の重複に加えて、メニュー所要時間を考慮した重複予約を防げる状態です。既存テストが通り、予約登録の基本操作が壊れていないことを確認します。

### Step 3：LINE本番接続

何を作るか:
LINE公式アカウントのWebhookと本番接続し、署名検証、返信処理、環境変数設定、疎通確認手順を整えます。

完了条件:
HTTPS公開されたWebhook URLでLINEからメッセージを受け取り、実際のLINEトーク画面へ返信できます。トークンやシークレットは環境変数で管理され、GitHubに含まれません。

LINEから予約が入った時に、管理者のLINEまたは店舗用LINEグループへ通知する機能はStep 3以降の拡張対象です。現時点では、予約は管理画面で確認する設計です。本番運用では、管理者通知、エラーログ、通知失敗時の再送管理が必要です。

### Step 4：LINE上のボタン式予約

何を作るか:
Flex Messageまたはクイックリプライを使い、メニュー選択、日時選択、予約確認をボタン操作で進められるようにします。

完了条件:
お客様が自由入力をほとんど使わずに、LINE上で予約を完了できます。選択途中の状態管理ができ、入力ミスによる予約失敗が減っている状態です。

### Step 5：予約変更機能

何を作るか:
既存予約の日時・メニュー変更機能を作ります。LINE上と管理画面の両方から変更できる設計にします。

完了条件:
お客様がLINEで予約変更でき、店舗側も管理画面で変更できます。変更後も営業時間、定休日、重複予約、メニュー所要時間のチェックが有効です。

### Step 6：予約前のお知らせ通知

何を作るか:
LINE Push Messageで予約前のお知らせを本番送信します。送信済み管理、送信失敗ログ、再送方針を整えます。

完了条件:
予約前の指定時間にLINE通知が送られ、送信済みの予約には重複送信されません。送信に失敗した場合はログで確認できます。

### Step 7：管理画面の強化

何を作るか:
予約検索、顧客別履歴、予約詳細、予約変更、設定確認、スマホ表示の見直しを行います。画面文言の文字化けも解消します。

完了条件:
店舗担当者が日常業務で必要な予約確認、登録、変更、キャンセル、履歴確認を管理画面だけで行えます。スマホでも主要操作が読みやすく、操作しやすい状態です。

### Step 8：データ管理・バックアップ

何を作るか:
SQLiteデータのバックアップ、復元手順、CSV出力の運用、必要に応じたDB移行方針を整えます。

完了条件:
予約データを定期的にバックアップでき、障害時に復元手順を実行できます。バックアップ先と保持期間が明確です。

### Step 9：本番運用・デプロイ準備

何を作るか:
HTTPS環境、環境変数、起動方法、監視、ログ確認、障害時対応、LINE公式アカウント設定手順を整備します。

完了条件:
READMEまたは運用ドキュメントに従って本番環境へデプロイでき、再起動・ログ確認・バックアップ確認ができます。

### Step 10：v1.0完成確認

何を作るか:
実店舗に近いシナリオで、LINE予約、確認、変更、キャンセル、管理画面操作、通知、バックアップ、復元、テストを通しで確認します。

完了条件:
v1.0完成条件をすべて満たし、既知の重大不具合がなく、商品説明・導入手順・運用手順が揃っている状態です。

## 5. Step 2で実装すべき内容

Step 2では、LINE本番接続やFlex Messageはまだ実装しません。先に予約ルールを固めないと、LINE側の操作を作っても実運用で予約枠が崩れやすいためです。

Step 2の対象は以下です。

- 営業日設定: 営業する曜日を設定として扱い、定休日には予約できないようにします。
- 営業時間設定: 開始時刻と終了時刻を予約登録時のチェックに使います。
- 予約間隔: 15分、30分、60分など、予約を受け付ける時間単位を定義します。
- 予約可能枠: 営業日、営業時間、予約間隔、既存予約をもとに、予約可能な候補枠を算出できるようにします。
- メニュー所要時間との整合性: 30分メニュー、60分メニュー、90分メニューなどの所要時間を考慮し、終了時刻が営業時間を超えないようにします。
- 予約枠の重複防止: 同一開始時刻だけでなく、予約時間帯が重なるケースも防ぎます。例として、10:00から60分の予約がある場合、10:30から30分の予約は登録できないようにします。
- 管理画面からの設定確認: 店舗担当者が現在の営業日、営業時間、予約間隔、予約ルールを管理画面で確認できるようにします。

Step 2の完了条件は以下です。

- 予約ルールを扱う設定・サービス設計が整理されている。
- 営業日、営業時間、予約間隔をもとに予約可否を判定できる。
- メニュー所要時間を考慮した重複防止ができる。
- 営業時間終了をまたぐ予約を防止できる。
- 管理画面で現在の予約ルールを確認できる。
- 既存の予約登録、キャンセル、CSV出力、メニュー管理が壊れていない。
- `python -m compileall app ui` と `python -m pytest -q` が成功する。

## 6. リスクと注意点

- LINE公式アカウント設定が必要です。Messaging APIの有効化、Webhook URL、応答設定、チャネルアクセストークン、チャネルシークレットを正しく設定する必要があります。
- Webhook公開にはHTTPS環境が必要です。ローカル実行だけではLINE本番接続は完了しません。
- 個人情報を扱うためアクセス制限が必要です。顧客名、LINEユーザーID、予約日時、備考を扱うため、管理画面の認証と公開範囲に注意が必要です。
- 予約データのバックアップが必要です。SQLiteファイルが失われると予約情報も失われるため、定期バックアップが必要です。
- 無料ホスティングでは常時稼働に注意が必要です。スリープ、再起動、永続ディスクの有無、Webhook応答速度を確認する必要があります。
- スマホ画面の見やすさ確認が必要です。小規模事業者はスマホで予約管理する可能性が高いため、主要画面の表示確認が必要です。
- LINE APIの送信制限やエラー対応が必要です。Push通知や返信に失敗した場合のログと再送方針が必要です。
- 文字化けして見える文言は商品化前に修正が必要です。お客様向け・店舗向けの表示文言は信頼性に直結します。
- 予約ルールが曖昧なままLINE UIを作ると、実運用で重複予約や営業時間外予約が発生しやすくなります。
